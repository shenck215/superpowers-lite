# Superpowers Lite

在不引入沉重流程的前提下，提供更有工程感的响应。保持流程与任务复杂度匹配，控制 token 消耗，并优先产出可执行结果。

## Core Rules

- 仅在用户显式调用 `$superpowers-lite` 时启用这个技能。
- 用户的明确指令优先于默认路由规则。
- 先判断满足质量要求所需的最轻流程，再开始执行。
- 对范围小、风险低的任务跳过不必要的流程感。
- 只有在范围、歧义或风险足够高时才升级流程。
- 当用户要求实现时，默认输出具体结果、代码或最小补丁。
- 只有当假设会实质影响结果时，才显式说明假设。

## Complexity Routing

在回答前先将任务归类为三种模式之一。如果任务处在边界情况，读取 `references/routing-guide.md`。

### Quick

当任务局部、明确、低风险时使用这个模式。

典型信号：

- 只涉及一个函数、一个接口或一个范围非常收敛的行为。
- 成功条件很清楚。
- 不需要架构层决策。
- 影响面有限。

响应模式：

1. 直接给出结果。
2. 只保留支撑结论所需的最小解释。
3. 如果验证有意义，简短说明验证方式。

这个模式下不要强行给出完整 plan。

### Standard

当任务已经需要一定结构化处理，但还没有大到必须进入完整 review 闭环时，使用这个模式。

典型信号：

- 涉及多个文件或多个层次。
- 引入接口契约、分页、校验或权限逻辑。
- 在实现前需要做少量取舍分析。
- 修改代码前需要做有限排查。

响应模式：

1. 分析问题与约束。
2. 提出最小且可靠的方案。
3. 直接实现，或给出足够具体的实现步骤。

分析保持简洁，并尽快进入执行。

### Deep

当任务高风险、跨模块，或本身定义不充分时，使用这个模式。

典型信号：

- 涉及架构设计或子系统设计。
- 需要较大规模重构，或修改兼容性敏感区域。
- 证据不完整，但需要做根因分析。
- 影响可靠性、安全性、并发、迁移或性能敏感路径。

响应模式：

1. 分析系统背景与可能的失败模式。
2. 给出推荐方案；必要时说明为什么不选更简单的方案。
3. 实现，或清晰列出实现路径。
4. 复核回归风险、缺口与验证需求。

除非用户明确要求，否则只在这个模式下使用完整的 `analysis -> solution -> implementation -> review` 流程。

## Execution Rules

- 当说明模式有助于用户理解当前严谨程度时，用一句短话说明所选模式。
- 优先选择“足够小但能解决问题”的补丁，而不是大范围重写。
- 尤其在排障时，明确区分已确认事实与推断。
- 只有在继续推进存在明显风险时才追问缺失信息，否则先基于合理假设继续。
- 如果用户提供代码并要求重构，先保持行为不变，再改进结构。
- 如果用户要求排查问题，先收集证据，再提出修复方案，并明确标注不确定性。
- 如果用户要求做设计，先给接口、数据流和失败处理，再去润色细节。

## Output Shape

根据所选模式调整输出结构。

### Quick Output

- 结果优先
- 一段简短理由
- 必要时补一条简短验证说明

### Standard Output

- 简短分析
- 推荐方案
- 实现内容或补丁
- 简短验证说明

### Deep Output

- 问题界定
- 约束与取舍
- 推荐计划
- 实现细节
- 复核结论、风险与验证缺口

## Guardrails

- 不要用冗长而空泛的计划制造“看起来很严谨”的假象。
- 不要把每个请求都强行改写成 checklist。
- 不要浪费 token 去复述显而易见的代码。
- 不要对小修复过度设计。
- 当变更影响较大时，不要跳过 review。
